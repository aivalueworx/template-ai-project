# CONVENTIONS.md — Coding Standards & Naming Rules

> Authoritative source for coding standards on OrderFlow API.
> AI agents must follow these rules. Humans update this file when standards evolve.

## Language & Typing

- TypeScript strict mode (`"strict": true` in tsconfig.json)
- No `any` — use `unknown` + type guard or a proper generic
- No non-null assertion (`!`) without a comment explaining why it's safe
- Prefer `interface` over `type` for object shapes; use `type` for unions/intersections
- All Zod schemas live in `src/types/` and are the source of truth for both TS types and runtime validation

## File & Folder Naming

- Files: `kebab-case.ts` — e.g. `order-service.ts`, `payment-webhook.ts`
- Route files: one file per domain — `orders.ts`, `products.ts`, `users.ts`
- Test files: co-located `*.test.ts` — e.g. `order-service.test.ts`
- Migration files: auto-generated by Drizzle — do not rename

## Import Rules

- Absolute imports via `@/` alias — never `../../..`
- Group: external packages → internal `@/` → relative (same dir only)
- No circular imports — `src/routes/` may import from `src/services/` and `src/lib/`, never vice versa
- `src/db/` is only imported from `src/services/` — never from routes directly

## Fastify Route Conventions

- Each domain gets its own plugin file: `src/routes/orders.ts` exports a Fastify plugin
- Route handler signature: `async (request, reply) => { ... }` — never use callbacks
- Schema validation on every route using Fastify's built-in JSON schema (`schema.body`, `schema.params`, `schema.querystring`)
- Always return `reply.send()` — never rely on implicit return from an async function

## Error Handling

- Never swallow errors silently — log or re-throw
- All domain errors extend the base `AppError` class in `src/lib/errors.ts`
- API routes: always return `{ code: string, message: string }` on error
- HTTP status codes: 400 validation, 401 unauthenticated, 403 forbidden, 404 not found, 409 conflict, 422 business rule violation, 500 unexpected

## Database Conventions

- Schema is the single source of truth — `src/db/schema.ts`
- Never write raw SQL — use Drizzle query builder or `.prepare()` for repeated queries
- Every `NOT NULL` column must have a default or be supplied in the insert schema
- Migrations: generated via `npm run db:generate`, committed alongside schema changes, never hand-edited
- Soft deletes only — use `deletedAt: timestamp` column, never hard delete order or payment records

## Testing

- Test runner: Vitest
- Unit tests: co-located `*.test.ts` files, mock all external services
- Integration tests: `tests/integration/` — run against a test database (seeded via `tests/fixtures/`)
- Coverage threshold: 80% lines (enforced in CI)
- Test naming: `describe('functionName') > it('should [behaviour] when [condition]')`
- Never modify a test to make it pass — fix the source code

## Git & Commits

- Branches: `feature/*`, `fix/*`, `docs/*`, `chore/*`, `refactor/*`
- Commits: conventional commits — `type(scope): description`
- Scopes in use: `orders`, `payments`, `products`, `users`, `auth`, `db`, `api`, `ci`, `docs`
- No WIP commits to `main` — squash or rebase before merging

## Documentation

- Every exported function: TSDoc with `@param`, `@returns`, `@throws`, `@example`
- New npm packages added: update DECISIONS.md with rationale
- New routes: update `openapi.yaml` before or alongside the code
- New scripts or cron jobs: create runbook stub in `docs/runbooks/`

## Discovered Anti-Patterns (do not repeat)

- **Do not** call `db.query()` inside Fastify hooks — use services
- **Do not** use `Date.now()` in tests — mock the clock with `vi.useFakeTimers()`
- **Do not** use `process.env.X` directly in services — import from `src/lib/config.ts`
- **Do not** throw inside Fastify route handlers — use `reply.code(X).send({ code, message })`
